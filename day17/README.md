# Day 17: Reservoir Research

Yikes, another one that took me many days of troubleshooting and generally just figuring out exactly what I needed to do. This was another physics simulator of water falling. I had to reach back to [2022 Day 14](https://github.com/AgroDan/aoc2022/tree/main/day14) to get some idea of how I should be handling this from a gravity behavior point of view. That challenge had to deal with falling sand which piled up in the shape of a triangle, which in hindsight was probably a bit harder to work with. But regardless this challenge was no picnic either. For some reason I just couldn't wrap my head around the best way to handle the whole "overflow" concept, where (as it showed in the example data) if water is considered "active" or "flowing" (`|`) rather than "settling" (`~`) then there different conditions for it to operate based on that. It was funky, but I went by this notion of "Ticks," whereby each "tick" was one movement box for every movable item on the runemap. I just had to determine what was considered moveable.

I tracked it by creating a slice containing every coordinate on the runemap that was considered "moving water," or essentially `|` runes. I would loop through every coordinate this pertained to and decide if this would move in any direction. This would wind up being incredibly inefficient because the moving water boxes added up considerably and it would start making repetitive decisions, checking to see if a moving water run had a similar `|` rune beneath it to determine if it would propogate. As the water continued to fall, the calculations became too much to the point that it just wouldn't finish in a reasonable amount of time.

Ultimately, I decided to instead maintain a slice of "falling points", or essentially water that had not yet settled. It would simply leave a trail of `|` runes behind it as it fell. So for each tick, each falling point would check the rune exactly one unit below it and check if it's a `.` rune. If so, then it just changes that rune to `|` and updates the falling point to the unit below itself. That's the end of that tick.

The real complexity however came in the concept of hitting a floor. When water would hit a stopping point (for the sake of simplicity, I'll say a stopping point is just a wall or a `#` rune), it would then need to spread. The spreading method I went with was a breadth-first-search algorithm, but only with the concept of spreading left and right from the point we're checking. However, the BFS algorithm was looking for two outcomes in either direction. Either we would find a "dropoff" point, or basically any point where the character _below_ it was a `.`, or we'd hit right up against a wall. Whichever came first is where it would end. Either the `bfsSpread()` function would return at most `2` dropoff points, or a wall. I'll try to illustrate it here:

```
...|...
...|..#
..#####
.......
.......
```

In this case, water is flowing downward until it hits a solid floor. At this point it can't move down for the next tick, so it instead calls upon the `bfsSpread()` function. This then looks left and right from that lower `|` rune and checks if there's a dropoff point:

```
...|...
..|||.#
..#####
.......
.......
```

Now in the above, it checks the surrounding `|` runes for dropoff points. In this case, there are none. The loop continues.

```
...|...
.D||||#
..#####
.......
.......
```

Continuing in both directions, to the right we don't find another dropoff point, but we do find a wall so we can't continue further. On the left side however, we _do_ find a dropoff point. This point is now marked as another "falling point" in the larger slice which keeps track of these. The new falling point is tracked now through the same separate rules.

I simplified the above mostly because I separated it into two functions, one that checks for dropoff points (`bfsSpread()`), and another that fills the remainder of the area with appropriate runes (Specifically `fill()`). If we are "contained," meaning `bfsSpread()` returned no dropoff points, then we should fill with `~` characters to the left and right, indicating this water has settled.

What took me the most time here was finding edge cases that wouldn't compute. The good part of this is that I was able to print out the map in full screen so I could see weird behavior as it happened. It was relatively easy to find anomalies due to the fact that it was naturally easy to see how the water should fall and spread. The tricky part was finding the logic that allowed for certain edge cases to appear.
